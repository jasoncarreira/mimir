# Upgrade to 0.6.5 — channel memory + memory hygiene

You just upgraded mimir from `{from}` to `{to}`. This release tightens the
channel-memory filing rules and adds a `memory-hygiene` scheduled scan. Do a
one-time pass now rather than waiting for the next weekly tick.

1. **Re-read the channel-memory rule.** Open `memory/core/60-filing-rules.md`
   and the `memory/channels/<id>/` section: it's small curated facts for a
   *real* channel (operator name, prefs, recurring patterns) — not a session
   journal. Only top-level `*.md` are auto-loaded, lexically, capped at ~8 KB;
   synthetic channels (`scheduler:*`, `poller:*`) and topic-shaped subfolders
   never load. Episodic detail belongs in SAGA, not channel files.

2. **Audit your channel memories against that.** Skim `memory/channels/*`:
   flag oversized channel dirs (approaching/over the ~8 KB cap), synthetic-
   channel notes that read like runbooks/wiki material, and anything episodic
   that should be SAGA instead. Flag, don't delete — deletions under
   `/mimir-home` are escalate-first.

3. **Run the memory-hygiene scan if available.** If a `memory-hygiene` tick
   (or `prompts/memory-hygiene.md`) exists, run that deterministic scan now to
   surface channel-memory bloat, stale/resolved `memory/issues/*`, missing
   `<!-- desc: -->` lines, and obvious misfiles. If it isn't present yet in
   this version, do the manual channel-memory review in step 2 instead.

4. **Record follow-ups; don't mutate protected surfaces.** File Chainlink
   issues for real cleanup, or open `open_proposal` / `submit_proposal` PRs for
   `memory/core/*` or `prompts/*` changes. Do not edit protected surfaces
   directly.

5. **Report once.** If there's anything operator-actionable, send a one-line
   digest to the operator alert channel (pass an explicit `channel_id` — this
   is a non-interactive turn). Otherwise end the turn silently after writing
   durable follow-ups.

This is a one-time migration nudge; the weekly `memory-hygiene` tick handles
this going forward.
