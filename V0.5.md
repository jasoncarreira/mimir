# Mimir v0.5 — workspace monorepo + bundled saga

**Goal:** end the cross-repo coordination tax between mimir and msam2 by merging the MSAM source tree into mimir as a workspace package (renamed `saga` along the way — §6), then close the operator-UX gap so `mimir setup && mimir run` brings up a working agent + memory pair without separate install ceremony.

**Frame:** v0.4 made mimir self-aware about resources (cost, plan windows, subagent spend). The next blocker on velocity is structural — every PR that touches the mimir/MSAM boundary today is a two-repo coordination dance, and every new operator who installs mimir hits a wall when MSAM isn't running. v0.5 makes both go away. The package gets renamed `msam` → `saga` (§6) along the way — Norse poet-goddess of history, pairs with mimir at the pantheon level, "saga" already carries the English meaning "ongoing chronicle" so the name self-explains. Throughout this doc, references to "msam" or "MSAM" describe the *current* state (msam2 repo, msam package); references to "saga" describe the post-rename state. Where the timing matters (e.g., the merge commit lands under `saga/` but the inner Python package is still named `msam` until §6 finishes), the text is explicit.

## Pieces (high-leverage to small)

1. [Workspace monorepo merge](#1-workspace-monorepo-merge) — filter-repo msam2 into mimir at `/saga/` (the package gets renamed from `msam` to `saga` as part of this work; see §6), set up uv workspace, archive msam2. The structural piece.
2. [Bundled saga lifecycle](#2-bundled-saga-lifecycle) — `mimir setup` generates a shared API key into both configs, `mimir run` supervises saga as a child process, opt-out via `MIMIR_SPAWN_SAGA=false` for external-saga deployments.
3. [Integration bench harness](#3-integration-bench-harness) — end-to-end benches that exercise saga through mimir's actual call sites, complementing saga-only synthetic benches.
4. [muninnbot pin + transition note](#4-muninnbot-pin) — pin muninnbot's docker to a known msam2 source commit before archiving, so its build keeps working through the deprecation window.
5. [Doc cleanup](#5-doc-cleanup) — V0.4.md path refs, SPEC.md cross-refs, README, FUTURE_WORK §11 update.
6. [Rename: msam → saga](#6-rename-msam--saga) — the package is renamed as part of v0.5, in a focused follow-up commit on top of the merge. Touches the saga package's internal imports, mimir's `MsamClient` / config / env vars, and event type names.

**Deferred to v0.6+:**
- Per-channel persona — design from v0.4's deferred list, ~50 LOC change in `load_core` to take channel_id. Independent of the merge work; pick up when feature priorities call for it.
- Temporal index for retrieval (FUTURE_WORK §3.1) — dominant probe-type miss in bench data; needs saga-side schema work that's much easier post-merge.
- Reply / repost graph (FUTURE_WORK §3.3) — weakest probe types; similar dependency pattern.
- Belief-revision / claims-audit reflection track (FUTURE_WORK §8.3 partial) — extends the v0.4 reflection skill once the operator and agent have settled patterns from a few weeks of running it.

### Constraint: saga stays standalone-runnable

Operators / contributors must be able to:

```bash
# From the merged repo (post-rename — see §6)
cd saga && uv run python -m saga.benchmarks.longmemeval.run_eval ...
# OR install saga in isolation (no mimir checkout)
uv pip install saga
```

…without mimir being importable. Concrete requirements:

1. **`saga/pyproject.toml` does NOT list mimir as a dependency.** Workspace ergonomics give both packages a shared venv when developing in this repo, but published metadata stays clean.
2. **The saga Python package never imports mimir.** Including transitively. `import saga` from a venv with no mimir installed must succeed.
3. **saga-only bench harness** (`saga/saga/benchmarks/longmemeval/`) stays inside the saga package — it tests saga in isolation and shouldn't carry mimir along.
4. **Cross-package integration benches live OUTSIDE the saga package** at `benchmarks/longmemeval_via_mimir/` (workspace root). They import both, but saga doesn't import them.

### CI validates standalone-ness

A separate CI job that's NOT a workspace install:

```yaml
- name: saga-standalone
  run: |
    cd saga
    uv venv .venv-standalone
    uv pip install --python .venv-standalone/bin/python -e .
    .venv-standalone/bin/python -c "import saga; print('OK')"
    .venv-standalone/bin/python -m pytest tests/
```

If this job fails, mimir leaked into saga's deps. Fix at the boundary, don't paper over.

## 2. Bundled saga lifecycle

**Goal:** new operator runs `mimir setup --home X && mimir run --home X` and gets a working agent + memory pair, no separate saga install ceremony. External saga deployments still work via `SAGA_ENDPOINT` override.

(All references in this section assume §6 has landed and the package is named `saga`. If reading post-merge but pre-rename, mentally substitute `msam` → `saga` and `MSAM_*` → `SAGA_*`.)

### Setup-time integration

`mimir setup` post-merge gains saga-aware behavior:

1. Detects whether `saga` is importable in the current Python env (it always will be in the workspace install).
2. Generates a single API key (`secrets.token_urlsafe(32)`).
3. Writes the key to mimir's `<home>/.env` as `SAGA_API_KEY=...`.
4. Writes a generated `<home>/saga.toml` derived from `saga/saga.example.toml` with mimir-prod overrides:
   - `[server].api_key = <generated>`
   - `[server].port = <auto-pick free port>` (default 3002, fall back to next free)
   - `[storage].db_path = <home>/.mimir/saga.db`
   - **`[retrieval].enable_contextual_rewrite = true`** — mimir already passes `context=` on every `/v1/query`; flipping the rewrite on means short referential queries ("yes, look for that") get resolved before retrieval. No-op when context is empty, so safe to leave on.
   - Otherwise inherits saga's canonical defaults (two_tier on, supersedes_demotion on, P12 query expansion on, confidence gating with `low` floor, P30 missing-ref pivot, spreading activation on). These are saga's `_DEFAULTS` per its `config.py`; the example.toml documents them.
5. Writes mimir's `SAGA_ENDPOINT=http://localhost:<chosen port>` matching.

Existing keys preserved on re-run, like the existing MIMIR_API_KEY behavior. Existing `<home>/saga.toml` files preserved too — operator can edit freely without setup clobbering.

The contextual_rewrite LLM call uses saga's `[retrieval_v2]` LLM config (falls back to `[llm]`). Mimir's setup should make sure at least one LLM is configured for saga if rewrite is on; otherwise the rewrite logs a warning and falls through to the original query (still functional).

### Run-time supervisor

`mimir run` post-merge gains a saga child-process supervisor when `MIMIR_SPAWN_SAGA=true` (default `true` if `saga` is importable; default `false` if not).

```python
# mimir/saga_supervisor.py — new module
class SagaSupervisor:
    """Forks saga.server, monitors, restarts on crash with backoff,
    pipes stderr into mimir's logger, graceful-shuts on parent signal."""

    async def start(self) -> None:
        # spawn, wait for /v1/health to return 200
        ...

    async def _monitor(self) -> None:
        # poll process; on exit, restart with exponential backoff
        # cap at N restarts in M minutes; past that, log critically
        # and stop trying (operator's problem)
        ...

    async def stop(self) -> None:
        # SIGTERM, wait for graceful shutdown, SIGKILL if needed
        ...
```

Supervisor lifecycle hooks into mimir's existing aiohttp app startup/shutdown handlers in `server.py`.

### Health check + visibility

- Startup log: `"saga supervisor: spawned PID 12345 on port 3002, /v1/health OK"` or `"SAGA_ENDPOINT=external; not spawning"`.
- New event types: `saga_supervisor_spawned`, `saga_supervisor_restarted`, `saga_supervisor_exited`. Surfaced in algedonic block via `feedback.py` rules so the agent sees if saga is unhealthy.
- `mimir stats` gains a saga line: process status, uptime, restart count.

### Opt-out paths

- `MIMIR_SPAWN_SAGA=false` — disable spawn entirely.
- `SAGA_ENDPOINT` set to non-localhost → automatically disable spawn (operator pointing at external saga).
- `mimir setup --no-saga` flag → setup skips saga config generation.

### Tests

- `tests/test_saga_supervisor.py`:
  - Spawn happy path (mock subprocess)
  - Health-check timeout → supervisor surfaces error
  - Crash + restart with backoff
  - Graceful shutdown on parent stop
  - External saga endpoint → supervisor doesn't spawn

### Effort

- `SagaSupervisor` module: ~150 LOC
- Setup integration: ~80 LOC (key generation, saga.toml writer)
- Tests: ~150 LOC (subprocess mocking is the bulk)
- `feedback.py` rule additions: ~10 LOC

**Total: ~390 LOC. ~2 focused days.**

## 3. Integration bench — mimir-side hooks on the existing LongMemEval harness

**Goal:** make saga's existing LongMemEval bench measure retrieval quality *through* mimir's actual call sites, not just saga-in-isolation. So when v0.6+ retrieval changes ship, we know whether the win is still a win after mimir's caching, contextual-rewrite, credit-pass, and session-boundary lifecycle interact with it.

### What's already there (post-merge inheritance)

saga's bench at `saga/saga/benchmarks/longmemeval/` (inside the Python package, runnable standalone) is mature:

- LongMemEval `S` (500 questions). Reader: MiniMax-M2.7. Judge: gpt-4o. Consolidation: gpt-5.4-nano.
- `run_eval.py` orchestrates per-question sqlite DBs (created in a work dir, destroyed after each question unless `--keep-dbs`).
- Variance characterized: ±1.3pp overall (N=500), per-subtype noise floors documented in `BENCHMARK-RESULTS.md`. Don't ship on sub-noise-floor wins.
- Canonical baseline: `msam_p30_minimax_v3` at 0.784. Best to date: `msam_p9_minimax_v2` at 0.796 (older codebase) and `msam_p12_canon_v1` at 0.792 (P12 query expansion, recently shipped).

This harness is the source of truth for retrieval quality. Don't replace it; extend it.

### What's invisible to saga-in-isolation benches

LongMemEval today drives saga directly via the Python API or `/v1/query`. It doesn't exercise:

- **Cache interaction.** Mimir's prompt cache changes which atoms get re-fetched turn-to-turn; whether a retrieval improvement compounds with cache hits or fights them is invisible at the saga layer.
- **Contextual rewrite firing rate.** With `enable_contextual_rewrite = true` (now the v0.5 default), short referential queries get rewritten via LLM. Bench shows the rewrite firing on the appropriate fraction of LongMemEval probes, the rewrites' quality, and the resulting end-to-end delta.
- **Mark-contributions credit pass.** Mimir's `mark_contributions` after each agent reply tags atoms that contributed; that signal feeds future retrieval ranking. End-to-end bench shows whether the credit signal is actually helping over multi-question runs.
- **Session boundaries.** Which `session_id` is in flight (per the `e88d458` schema migration) determines the credit-pass scope. End-to-end bench measures within-session vs. cross-session recall.
- **`recent_session_boundaries` / `most_retrieved_atoms` (P45) usage.** These are mimir-side prompt sections. Whether populating them improves end-to-end task accuracy is a question the saga-only bench can't answer.

### Harness shape

The integration bench lives at the **workspace root**, not inside `saga/`, so the saga package stays mimir-independent:

```
mimir/                                  (workspace root)
├── saga/saga/benchmarks/longmemeval/   # existing — saga-direct, unchanged.
│   ├── run_eval.py                     #   Imports saga only.
│   └── ...                             #   Runnable standalone:
│                                       #     cd saga && uv run python -m \
│                                       #       saga.benchmarks.longmemeval.run_eval
│
└── benchmarks/longmemeval_via_mimir/   # NEW — cross-package integration.
    ├── runner.py                       #   Imports both mimir and saga.
    ├── route.py                        #   Spawns supervised saga, boots mimir's
    ├── score.py                        #   BenchBridge agent in-process, adapts
    │                                   #   LongMemEval probes → channel messages.
    │                                   #   Reuses saga's scoring via score.py.
    └── README.md
```

Why outside `saga/`: if `longmemeval_via_mimir/runner.py` lived under the saga package and imported mimir, then importing the saga package transitively would fail in any venv where mimir isn't installed. Standalone saga ops would silently break. Living at workspace root makes the dependency direction obvious: integration benches use both packages; neither package depends on the integration tree.

The runner uses mimir's existing `BenchBridge` (designed for this) so bench probes hit the same dispatch path as a live Slack/Discord message — pre_message_hook fires (queries saga), the agent reads the assembled prompt (with context, plan windows, etc.), assembles its response, post_message_hook fires (mark_contributions). All visible.

### Path-frequency telemetry

The handoff doc points at saga's `saga.retrieval` Python logger (renamed from `msam.retrieval` in §6) which emits one tagged line per LLM-path decision (`path=combined`, `path=rewrite_only`, `path=hyde_gated fired=...`, `path=subatom_beam atoms=N`, `path=triple_augment_v2 atoms=N`). v0.5 wires this into mimir's logging setup so:

- The bench's stderr capture shows path frequencies per LongMemEval run.
- `mimir stats` gains a path-frequency snapshot for production use.
- The algedonic block surfaces a warning if (e.g.) HyDE keeps firing in a config that didn't enable it — a misconfiguration smoke alarm.

Per the handoff's first-30-minutes guide (preserved in `saga/HINDSIGHT-IDEAS.md` / `saga/NEXT-EXPERIMENTS.md` post-merge):

## 6. Rename: msam → saga

**Goal:** the package becomes `saga` end-to-end. Decided after a session of brainstorming; rationale in V0.5.md commit history. Saga (Norse poet-goddess of history) pairs with mimir at the Norse-pantheon level and carries the natural English meaning "ongoing chronicle" — fits a memory store that accumulates and periodically consolidates.

The rename ships as part of v0.5, in two focused commits on top of the structural merge (§1) so each piece is reviewable on its own:

### Commit A: rename inside the saga package

After §1's merge has landed, the workspace member is named `saga/` but the Python package directory inside is still `msam/` and the package's internal imports / pyproject.toml metadata still say `msam`. Commit A finishes the rename within the saga package only:

```bash
# Inside the saga workspace member
cd saga

# 1. Rename the Python package directory.
git mv msam saga

# 2. Update pyproject.toml: name = "saga", any tool.* references.
sed -i '' 's/name = "msam"/name = "saga"/' pyproject.toml
sed -i '' 's/\[tool\.msam/[tool.saga/g' pyproject.toml  # if any

# 3. Sed import paths inside the package source.
find saga -name '*.py' -exec sed -i '' \
    -e 's/^from msam\./from saga./g' \
    -e 's/^from msam /from saga /g' \
    -e 's/^import msam$/import saga/g' \
    -e 's/^import msam\./import saga./g' \
    -e 's/from msam\.\([a-zA-Z_]*\)/from saga.\1/g' \
    {} \;

# 4. Update doc paths inside saga/ that reference msam (BENCHMARK-RESULTS,
#    NEXT-EXPERIMENTS, README, etc. all reference msam/ paths in examples).
```

**Event type renames** (the load-bearing decision):

Events emitted by mimir into `events.jsonl` that contain "msam" in their type:

- `msam_synthesis_empty_window` → `saga_synthesis_empty_window`

These are committed contracts with anyone grepping `events.jsonl` — including the introspection skill and feedback.py's classifier rules. Renaming breaks anyone who's grepping logs. **The clean break is fine here** — events.jsonl is bounded by `MIMIR_MAX_EVENTS` (default 5k, hard cap 50k; v0.4), so historical log entries with the old names rotate out within ~10k events; v0.5 ships an updated `feedback.py` rule table that classifies the new names; introspection skill examples get updated. No backwards-compat aliasing.

**Backwards-compat for env vars** (one cycle):

Operators have `MSAM_ENDPOINT` and `MSAM_API_KEY` set in their `.env` files from v0.4. Breaking that on upgrade is a migration tax. The right call: read `SAGA_ENDPOINT` first, fall back to `MSAM_ENDPOINT` with a deprecation log line. Same for the API key. Drop the fallback in v0.6.

```python
# Config.from_env():
saga_endpoint = _env("SAGA_ENDPOINT") or _env("MSAM_ENDPOINT")
if not _env("SAGA_ENDPOINT") and _env("MSAM_ENDPOINT"):
    log.warning(
        "MSAM_ENDPOINT is deprecated; rename to SAGA_ENDPOINT in .env. "
        "Compatibility fallback will be removed in v0.6."
    )
```

The `MIMIR_API_KEY` regenerate flow (v0.4 §6) doesn't touch `MSAM_API_KEY` / `SAGA_API_KEY`, so the operator's existing key stays valid through the rename.
