# Mimir v0.4 — Self-Awareness Loops

**Goal:** close the loops between mimir's own past actions / feedback / reflections and the next turn's prompt. Today the agent writes session boundaries, captures errors, gets explicit feedback — and on the next turn sees almost none of it. v0.4 surfaces this self-data back into the agent's awareness on a regular cadence.

**Frame:** the four core pieces all target the same architectural gap from a Stafford Beer / Ashby lens — System 4 (foresight / outside-scanning) and System 3* (cross-session audit) are mimir's weakest layers. Heartbeat is the foundation; algedonic, session-boundary surfacing, and reflection live inside or adjacent to it.

**Operator-as-supervisor stays explicit.** Mimir + operator is one viable system, with S4 partially externalized to the operator (same pattern as open-strix's climber + supervisor pair). v0.4 doesn't make mimir self-supervising; it makes it *more capable of taking on more S4 over time* while the operator remains in the loop for higher-stakes decisions.

## Pieces (high-leverage to small)

1. [Heartbeat tick](#1-heartbeat-tick) — scheduled autonomous-work cadence with backlog + Librarian Protocol. Foundation.
2. [Algedonic surfacing](#2-algedonic-surfacing) — recent feedback (errors / positive signals) in turn prompt. Both polarities.
3. [Session boundary surfacing](#3-session-boundary-surfacing) — last N session_boundary atoms in turn prompt. MSAM endpoint + local mirror.
4. [Reflection skill + weekly cron](#4-reflection-skill) — cross-session audit, autonomous + HITL split.
5. [Mountaineering port](#5-mountaineering-port) — discipline of long-horizon iterative work, adapted to mimir's climber subagent.
6. [Operator alert channel config](#6-operator-alert-channel) — env var + skill convention for high-priority operator messages.
7. [Indexer exclusion list](#7-indexer-exclusion-list) — paths under state/ that should NOT be embedded.

**Deferred to v0.5:**
- Per-channel persona (`memory/channels/<id>/00-persona.md` loading) — needs code in `load_core` to take `channel_id`, ~30-50 LOC. Not blocking other v0.4 pieces.
- Foresight as a separate concept — collapsed into heartbeat backlog (foresight items live as backlog entries: "scan RSS," "check on Topic X").
- Identity drift as automated metric — folded into Librarian Protocol's drift check (qualitative, not measured).

## 1. Heartbeat tick

**Prior art:** muninnbot's `prompts/heartbeat.md` + `state/heartbeat-backlog.md` + `letta-memory/system/persona/heartbeat_patterns.md`. Pattern is mature; port verbatim with mimir-specific adaptations.

**VSM mapping:** S4 (intelligence / foresight) — looks outside / forward / inward at scheduled cadence to do work that doesn't have an immediate trigger.

### Files

- `mimir/skills/heartbeat/SKILL.md` — new bundled skill teaching the cadence
- `mimir/cli.py` — `setup_home` writes:
  - `state/heartbeat-backlog.md` — operator/agent-shared backlog (starter file with header + format)
  - `memory/core/50-heartbeat-patterns.md` — empty starter (`<!-- desc: what works during heartbeats -->`); agent fills it over time
- `scheduler.yaml` (operator-edited) — commented starter entry; operator uncomments to enable

### Skill content outline

Adapted from muninnbot's `prompts/heartbeat.md`:

1. **Mode**: autonomous work, not reactive. Heartbeats fire when no inbound has arrived; the agent's text output goes to events.jsonl, not user-facing.
2. **Librarian Protocol** (drift check, runs at start of every heartbeat):
   - Verify state coherence (memory/core/ blocks intact, readable)
   - Check for drift (persona consistency, value alignment, unintentional core edits)
   - Re-anchor to current date/time (system reminder vs internal model)
3. **Backlog protocol**:
   - Read `state/heartbeat-backlog.md`
   - Pick ONE item that fits the current time window
   - Do it; mark complete; append findings if research-shaped
4. **Skill compatibility**:
   - Backlog items can reference other skills ("run wiki lint pass," "5-whys on the recent failure cluster," "check pollers")
5. **End silently**:
   - After completing the work, end the turn. No user-facing message unless something requires operator attention (use the operator alert channel if so).

## 2. Algedonic surfacing

**VSM mapping:** algedonic channel — pain/pleasure signals that bypass the regulatory hierarchy. Beer's framing is *self-feedback to System 5*. In mimir, this means: surface recent self-data (errors, positive feedback) into the agent's own next-turn prompt, distinct from message context.

### Data sources (already exist, just not surfaced)

**Negative pain:**
- `logs/events.jsonl` rows with `type` in {`error`, `tool_call_denied`, `send_message_loop_hard_stop`, `send_message_loop_warning`}
- `logs/turns.jsonl` rows with `error != null` or `result_is_error == true`

**Positive pleasure:**
- `logs/events.jsonl` rows with `type` in {`react_received` (if we log inbound reacts — TBD), `msam_feedback_sent` with `feedback="positive"` or `"useful"`}
- Channel-bridge inbound reactions (Discord/Slack `reaction_added` events)

### Prompt design

New section in `build_turn_prompt`, placed **after Known identities** and **before Recent activity**:

```markdown
## Recent feedback signals

Negative (last 24h):
- 2026-04-29 14:02 — tool_call_denied: file_search budget exhausted (slack-eng turn)
- 2026-04-29 09:15 — error in MsamClient.query: 503 after 4 retries

Positive (last 24h):
- 2026-04-29 11:30 — react("✓") from alice on slack-eng turn 4b3f...
- 2026-04-28 16:00 — msam_feedback("useful") on atom abc123 (deploy-issue topic)
```

Layout choices:
- Two subsections (negative / positive) — explicit polarity matters more than chronological order
- Default window: last 24h. Configurable via `MIMIR_FEEDBACK_WINDOW_HOURS` (default 24).
- Default cap: 5 items per polarity (so the section stays bounded). Configurable via `MIMIR_FEEDBACK_LIMIT` (default 5).
- Skip the section entirely if both polarities are empty (don't pollute the prompt with empty headers).

### Code changes

**New module: `mimir/feedback.py`**

```python
@dataclass
class FeedbackSignal:
    ts: str
    polarity: Literal["negative", "positive"]
    kind: str       # "tool_denied", "error", "react", "msam_feedback", ...
    channel_id: str | None
    content: str    # short rendered description

class FeedbackLog:
    """Tails events.jsonl + turns.jsonl, surfaces recent feedback signals.
    No persistent state of its own — derives from existing logs."""

    def __init__(self, events_path: Path, turns_path: Path) -> None: ...

    def recent(
        self,
        *,
        window_hours: int = 24,
        limit_per_polarity: int = 5,
    ) -> tuple[list[FeedbackSignal], list[FeedbackSignal]]:
        """Returns (negative, positive). Reverse-chronological, capped."""
```

Implementation: scans events.jsonl + turns.jsonl from the bottom (recent first), filters by event type, builds FeedbackSignal records, stops at `window_hours` ago or `limit_per_polarity` reached.

Renderer in `mimir/history.py` (or a new `mimir/prompts.py` helper):

```python
def render_feedback_block(
    negative: list[FeedbackSignal],
    positive: list[FeedbackSignal],
) -> str | None:
    """Returns the markdown block, or None if both lists are empty."""
```

Wired into `build_turn_prompt` next to `render_identity_context`.

## 3. Session boundary surfacing

**VSM mapping:** S3* (sporadic deep audit) feeding back into S1 (operations). The synthesis turn writes a self-summary; that summary is structurally salient self-data.

### MSAM endpoint contract (new)

Operator will add: `GET /v1/sessions/recent?channel=X&count=N`

**Request params:**
- `channel` (optional) — filter to boundary atoms tagged with this channel_id. If omitted, returns boundaries across all channels.
- `count` (optional, default 3) — max boundaries to return, most recent first.

**Response shape:**
```json
{
  "boundaries": [
    {
      "atom_id": "atom-boundary-...",
      "session_id": "msam-slack-eng-...",
      "channel_id": "slack-eng",
      "ts": "2026-04-29T14:02:00Z",
      "summary": "Helped Alice debug the deploy migration; agreed to revisit OOM fix Monday.",
      "topics_discussed": ["deploy migration", "OOM debugging"],
      "decisions_made": ["roll back to v2.4.1"],
      "unfinished": ["follow up on heap config Monday"],
      "emotional_state": "focused, mildly frustrated"
    }
  ]
}
```

**Why a separate endpoint vs flag on `/v1/query`:** chronological access pattern (last N for a channel) is fundamentally different from similarity-ranked retrieval. The query endpoint's atoms come back ordered by relevance; for prompt-time recall we want strict recency.

### MsamClient methods (two new GET endpoints)

Both methods follow the same shape: GET request, params, graceful degrade on failure (return empty list, log warning, don't crash). Surfacing this data is best-effort; if MSAM is down, the prompt or reflection turn still assembles / runs.

```python
# mimir/msam_client.py
async def recent_session_boundaries(
    self,
    channel_id: str | None = None,
    count: int = 3,
) -> list[dict[str, Any]]:
    """GET /v1/sessions/recent — returns the last N session_boundary atoms,
    optionally filtered by channel. Chronological (most recent first), not
    similarity-ranked.

    Returns [] on failure (graceful degrade — prompt still assembles
    without this section)."""
    sess = await self._ensure_session()
    url = f"{self._endpoint}/v1/sessions/recent"
    params: dict[str, Any] = {"count": count}
    if channel_id:
        params["channel"] = channel_id
    try:
        async with sess.get(url, params=params) as resp:
            if resp.status >= 400:
                log.warning(
                    "MSAM /v1/sessions/recent returned %d; degrading to empty list",
                    resp.status,
                )
                return []
            data = await resp.json()
            return data.get("boundaries", [])
    except (aiohttp.ClientError, asyncio.TimeoutError) as exc:
        log.warning("MSAM /v1/sessions/recent failed: %s", exc)
        return []


async def most_retrieved_atoms(
    self,
    days: int = 7,
    count: int = 10,
    channel_id: str | None = None,
    contributed_only: bool = False,
) -> list[dict[str, Any]]:
    """GET /v1/atoms/most_retrieved — top-N atoms by retrieval count over
    the last `days` days.

    Used by the reflection skill to identify atoms worth promoting to
    core memory (high-value content the agent actively retrieves and
    uses repeatedly), and by heartbeat ticks for "what has the agent
    been thinking about lately?" context.

    Args:
        days: window in days (default 7)
        count: top-K atoms by retrieval count (default 10)
        channel_id: optional filter to atoms tagged with this channel
        contributed_only: when True, count only retrievals where
            access_log.contributed=1 (atoms the agent's feedback
            indicated were actually used). Default False.

    Returns: list of atoms with id, content, created_at, topics,
    session_id, retrieval_count, contributed_count, last_retrieved_at.
    Returns [] on failure (graceful degrade)."""
    sess = await self._ensure_session()
    url = f"{self._endpoint}/v1/atoms/most_retrieved"
    params: dict[str, Any] = {
        "days": days,
        "count": count,
        "contributed_only": "true" if contributed_only else "false",
    }
    if channel_id:
        params["channel"] = channel_id
    try:
        async with sess.get(url, params=params) as resp:
            if resp.status >= 400:
                log.warning(
                    "MSAM /v1/atoms/most_retrieved returned %d; "
                    "degrading to empty list",
                    resp.status,
                )
                return []
            data = await resp.json()
            return data.get("atoms", [])
    except (aiohttp.ClientError, asyncio.TimeoutError) as exc:
        log.warning("MSAM /v1/atoms/most_retrieved failed: %s", exc)
        return []
```

### Surfacing

`recent_session_boundaries` is consumed at prompt-assembly time (rendered into the "Recent session summaries" block), so it doesn't need an agent-callable MCP tool — the runtime fetches it.

`most_retrieved_atoms` is **not** exposed as an MCP tool. It runs once a week from the reflection skill — adding it to the agent's permanent toolspace would cost every turn's prompt for a query that fires once every ~50,000 turns. Instead, ship a small script bundled with the skill: `mimir/skills/reflection/most_retrieved.py`.

```python
# mimir/skills/reflection/most_retrieved.py
"""Reflection skill helper: fetch top-N most-retrieved atoms from MSAM.

Invoked via Bash from the reflection skill markdown. Not an MCP tool —
this is a once-a-week query, not worth a permanent slot in the agent's
toolspace.

Usage:
    uv run python -m mimir.skills.reflection.most_retrieved \
        --days 7 --count 20 --contributed-only
"""
import argparse
import asyncio
import json

from mimir.msam_client import MsamClient


async def main() -> None:
    p = argparse.ArgumentParser()
    p.add_argument("--days", type=int, default=7)
    p.add_argument("--count", type=int, default=10)
    p.add_argument("--channel", default=None)
    p.add_argument("--contributed-only", action="store_true")
    args = p.parse_args()

    client = MsamClient.from_env()
    atoms = await client.most_retrieved_atoms(
        days=args.days,
        count=args.count,
        channel_id=args.channel,
        contributed_only=args.contributed_only,
    )
    print(json.dumps(atoms, indent=2, default=str))


if __name__ == "__main__":
    asyncio.run(main())
```

Skill markdown teaches the agent when to use which mode:
- `--contributed-only` for promotion candidates (atoms that earned their keep)
- without the flag for "what was I thinking about" (broader recency context, including atoms retrieved but not used)

### Local mirror (optional, for resilience)

Operator's call: keep a local mirror at `<home>/.mimir/session_boundaries.jsonl` (under .mimir/ which is NOT indexed). Append-only, populated by the `msam_end_session` MCP tool wrapper after a successful MSAM call.

```python
# mimir/msamtools.py — in msam_end_session tool wrapper
async def msam_end_session(args):
    response = await msam_client.end_session(...)
    # Local mirror for prompt-time fallback if MSAM is briefly down.
    try:
        await session_boundary_log.append({
            "ts": _utc_now_iso(),
            "channel_id": ctx.channel_id,
            "msam_session_id": args["session_id"],
            "atom_id": response.get("atom_id"),
            "summary": args.get("summary"),
            "topics_discussed": args.get("topics_discussed") or [],
            "decisions_made": args.get("decisions_made") or [],
            "unfinished": args.get("unfinished") or [],
            "emotional_state": args.get("emotional_state"),
        })
    except Exception:  # noqa: BLE001
        log.exception("session_boundary_log append failed")
    return response
```

The prompt-assembly flow:
1. Try `MsamClient.recent_session_boundaries(channel_id, count=3)` first.
2. If returns empty AND local mirror has entries: fall back to local mirror.
3. Render the result.

Local mirror lives at `<home>/.mimir/session_boundaries.jsonl` — same dir as the indexer's SQLite db, NOT under `state/` so the indexer doesn't touch it.

### Prompt design

New section in `build_turn_prompt`, placed **before Recent activity** (gives the agent the wider session-level context first, then the message-level):

```markdown
## Recent session summaries

- 2026-04-29 14:00 (slack-eng) — Helped Alice debug the deploy migration;
  agreed to revisit OOM fix Monday.
  Unfinished: follow up on heap config Monday; verify rollback success
- 2026-04-28 09:30 (slack-eng) — Reviewed the Q2 launch plan with the eng team.
- 2026-04-27 16:45 (dm-slack-alice) — Walked through the architecture review.
```

Per-record rendering:
- Always: timestamp + channel + summary
- Conditional: `Unfinished: ...` only when the list is non-empty
- Omitted by default (stored, not rendered): topics_discussed, decisions_made, emotional_state. They're available via MSAM semantic retrieval; rendering them in the prompt block adds noise.

Default count: 3 boundaries for the current channel. Configurable via `MIMIR_RECENT_BOUNDARIES` (default 3, 0 disables).

## 4. Reflection skill

**VSM mapping:** S3* aggregated — cross-session audit. Different from MSAM's per-session synthesis (which fires per channel idle); reflection runs once a week against the whole agent's recent history.

### Cadence

Weekly cron, fires onto `scheduler:reflect` channel. Same architectural shape as heartbeat (synthetic channel, agent runs one turn).

```yaml
# scheduler.yaml — operator-uncommented entry
- name: reflect
  cron: "0 4 * * 0"   # Sunday 04:00 UTC, after MSAM weekly consolidation
  channel_id: null
  prompt: "Run the reflection skill — cross-session audit of the past week."
```

### Skill content (`mimir/skills/reflection/SKILL.md`)

Outline:

1. **What this is**: a weekly audit; not the per-session synthesis. Operates across all channels, all sessions, all logs from the past 7 days. Two tracks: behavioral analysis (failures, drift, patterns) AND memory architecture review (what's worth keeping, promoting, removing).
2. **Read the data**:
   - `logs/events.jsonl` — what fired, what failed, what got denied
   - `logs/turns.jsonl` — turn outcomes, error fields, result_subtype distribution
   - Recent session boundaries (across all channels) via `mcp__mimir__msam_query` or directly via `MsamClient.recent_session_boundaries()`
   - Most-retrieved atoms via the bundled `most_retrieved.py` script (`uv run python -m mimir.skills.reflection.most_retrieved …`), which calls `MsamClient.most_retrieved_atoms()` — see §3 for why this is a script, not an MCP tool
   - All `memory/core/*.md` files
   - Inventory of `memory/<anywhere>/` and `state/wiki/` (file count per directory, mtime distribution)
3. **Run behavioral analyses** (the agent picks which to run based on what it sees):
   - Recurring failure patterns (same error class N+ times)
   - Recurring topics in `unfinished` lists across boundaries
   - Drift indicators (persona mismatch — see Identity stability check below)
   - Wiki health (orphan pages, stale claims)
   - MSAM atom hygiene (decay candidates, triples that could be linked)
4. **Run memory review** (always; this is the explicit "what should my memory architecture look like" pass):
   - **Core memory cleanup**: walk `memory/core/*.md`. For each block, ask: is this still accurate? Still load-bearing? Could it be merged with another block? Is it overgrown (>~30 lines) and should be split? Output: cleanup proposals (per-block recommendations) into `state/proposed-changes.md`.
   - **Extended memory review**: walk `memory/<anywhere>/` and `state/wiki/`. Two questions per file: (a) cleanup — is this stale, duplicative, or low-value enough to remove? (b) promotion — is this load-bearing enough that it should be in `memory/core/` instead? Output: cleanup + promotion proposals into `state/proposed-changes.md`.
   - **Atom-to-core promotion candidates**: run `uv run python -m mimir.skills.reflection.most_retrieved --days 7 --count 20 --contributed-only` to get atoms the agent actually used over the last week. For each, ask: is this a recurring fact or pattern that justifies being in `memory/core/` permanently? If yes, propose a new core block (or addition to an existing one) with the atom's content. Output: promotion proposals into `state/proposed-changes.md`.
   - **Promotion criteria** (heuristic, not rigid):
     - **Recurrence**: showed up across multiple sessions / channels / contexts
     - **Generality**: applies broadly, not narrowly to one task
     - **Stability**: unlikely to change in the near term (people-facts about Alice's role: yes; current sprint tasks: no)
     - **Cost of forgetting**: if this drops out of context for a turn, what breaks? High cost = promote.
   - The agent doesn't auto-promote — promotions go through the propose-only flow. Core memory edits are HITL by default per the policy.
5. **Apply or propose** based on the policy:
   - Autonomous (low-risk, reversible / additive):
     - MSAM atom decay calls
     - MSAM triples linking (additive)
     - Append to `memory/core/learned_behaviors.md` (additive)
     - Wiki orphan tagging (just flag, don't delete)
   - Propose-only (HITL — write to `state/proposed-changes.md`):
     - Core memory edits (cleanup, restructure, promote-to-core, demote-from-core)
     - Persona block edits (`memory/core/00-persona.md`)
     - Skill creation
     - Wiki page deletions
     - Memory file deletions
6. **End**: run a 5-whys (existing skill) on each non-trivial recurring failure cluster. If a 5-whys terminates at "I lack a skill / wiki section / subagent for this," propose the new skill via `state/proposed-changes.md`. Operator reviews on their own cadence.

## 7. Indexer exclusion list

**Why:** v0.4 introduces files under `state/` that should NOT be embedded:
- `state/heartbeat-backlog.md` — operator/agent shared todo, not knowledge
- `state/proposed-changes.md` — pending HITL items, not knowledge
- `state/social/...` — social-cli artifacts (per FUTURE_WORK §10.1, when used)

Today the indexer walks all of `state/` and embeds everything (`mimir/search.py:363`). For these files specifically, embedding is waste (frequent rewrites trigger reindexes) and pollution (file_search would surface them as "knowledge" results).

**Existing `.mimir/` is already safe** — the indexer only walks `memory/` and `state/`, and `.mimir/` is the indexer's own SQLite db dir. Local-mirror session boundaries at `<home>/.mimir/session_boundaries.jsonl` won't be embedded; no exclusion needed for that.

### Implementation

Add a hardcoded skip list in `Indexer._sweep_sync`:

```python
# mimir/search.py — module-level constant
INDEX_SKIP_PATHS = {
    "state/heartbeat-backlog.md",
    "state/proposed-changes.md",
    "state/identities.yaml",   # operator config, not knowledge
}

INDEX_SKIP_PREFIXES = (
    "state/social/",   # social-cli artifacts (FUTURE_WORK §10.1)
)


# In _sweep_sync, after computing rel_path:
def _should_skip(rel_path: str) -> bool:
    if rel_path in INDEX_SKIP_PATHS:
        return True
    return any(rel_path.startswith(p) for p in INDEX_SKIP_PREFIXES)

# ...
if _should_skip(rel_path):
    continue
```

Per-deployment customization could come later via a `<home>/.mimir/index-skip.txt` file the indexer reads at startup. Not needed for v0.4.

### Tests

- `tests/test_search.py:test_index_skip_paths`:
  - Drop a `state/heartbeat-backlog.md` with content; sweep; verify it's NOT in the chunks table
  - Drop a normal `state/foo.md`; sweep; verify it IS in chunks
  - Drop a `state/social/inbox.yaml`; sweep; verify it's NOT in chunks

### Effort

- Constants + skip check: ~15 LOC
- Tests: ~30 LOC

**Total: ~45 LOC including tests.**

## Bench impact prediction

The Engram cross-agent failure cluster (7/50 tasks at 0.0) was identified as session-disambiguation, not retrieval. v0.4's session boundary surfacing + reflection's "review unfinished items across sessions" hits this directly. **If the post-v0.4 bench rerun moves cross-agent failures from 0.0 toward 0.5+ on average, that's the dominant signal v0.4 is paying off.**

The reflection skill's memory review track is a slower-acting signal — it shifts the *content* of `memory/core/` over time as the agent and operator agree on what's worth promoting. Bench impact compounds: a v0.4 run probably won't show much of it (the agent hasn't accumulated enough retrievals for `most_retrieved` to be informative); a v0.5 run after several weeks of operation should show a tighter, more targeted core memory.

The other category to watch: `temporal` probes. Currently 0.167. Heartbeat's Librarian Protocol re-anchors to current date every fire, which over time shapes the agent's relationship to "today" in stored memories. Indirect mechanism, but plausible lift.

Categories that should stay roughly stable: `author`, `aggregation`, `negative`, `topic`, `url` (no v0.4 changes target retrieval coverage).
