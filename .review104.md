**Inline comments worth flagging:**

1. **CR2-#2 sub-item not addressed: wildcard CORS on `/chat/stream`.** `mimir/bridges/web_chat.py:168` still sets `Access-Control-Allow-Origin: *`. CR2-#2's fix shape said "Drop the SSE wildcard CORS or move it behind an opt-in env var." With the auth gate on, an attacker can't read the stream cross-origin without the key, so the practical risk is reduced — but a malicious page that already has the key (e.g. via XSS in another mimir surface, or a user pasting the key into a phishing page) could now exfiltrate the SSE feed cross-origin. Low-priority follow-up; flag, don't block.

2. **`?api_key=` in query string lands in aiohttp access logs.** server.py:131-137 falls back to query string for SSE clients that can't set headers. aiohttp's default access-log format includes the full request line with query string, so the secret ends up on disk wherever stdout is captured. Two reasonable mitigations: (a) install a custom access-log formatter that masks `api_key=` values, or (b) use a per-stream short-lived bearer that the HTML shell mints from an authed endpoint and discards on disconnect. Worth a follow-up issue, not a blocker for this PR.

3. **localStorage exposes the key to any third-party JS.** turn_viewer.html:217 + ops_dashboard.py inline JS load Chart.js + Inter from `cdn.jsdelivr.net` and `fonts.googleapis.com` without Subresource Integrity. If either CDN is compromised, the key leaks. Mitigation: add SRI hashes to the script/link tags, or self-host. Future hardening; not Pattern B scope.

4. **Exempt set exempts the path regardless of method.** server.py:93-97 + middleware:127. If someone ever adds `POST /turns` (e.g. for a server-side form), it would inherit the exemption. Cheap fix: store `(method, path)` tuples in the exempt set. Optional polish, not a real bug today.

5. **Threat-model question (not blocking):** the PR gates both reads (`/api/turns`, `/api/events`, `/api/ops`) and writes (`/event`, `/chat`) on the same key. CR2-#2 was framed as "read endpoints leak conversation history," but the original `/event` write was already gated. So a single shared key for read+write is the simplest defensible model and matches what shipped here. Worth confirming the operator handoff intended a single token rather than separate read/write keys.
