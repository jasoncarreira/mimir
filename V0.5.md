# Mimir v0.5 — workspace monorepo + bundled saga

**Goal:** end the cross-repo coordination tax between mimir and msam2 by merging the MSAM source tree into mimir as a workspace package (renamed `saga` along the way — §6), then close the operator-UX gap so `mimir setup && mimir run` brings up a working agent + memory pair without separate install ceremony.

**Frame:** v0.4 made mimir self-aware about resources (cost, plan windows, subagent spend). The next blocker on velocity is structural — every PR that touches the mimir/MSAM boundary today is a two-repo coordination dance, and every new operator who installs mimir hits a wall when MSAM isn't running. v0.5 makes both go away. The package gets renamed `msam` → `saga` (§6) along the way — Norse poet-goddess of history, pairs with mimir at the pantheon level, "saga" already carries the English meaning "ongoing chronicle" so the name self-explains. Throughout this doc, references to "msam" or "MSAM" describe the *current* state (msam2 repo, msam package); references to "saga" describe the post-rename state. Where the timing matters (e.g., the merge commit lands under `saga/` but the inner Python package is still named `msam` until §6 finishes), the text is explicit.

## Pieces (high-leverage to small)

1. [Workspace monorepo merge](#1-workspace-monorepo-merge) — filter-repo msam2 into mimir at `/saga/` (the package gets renamed from `msam` to `saga` as part of this work; see §6), set up uv workspace, archive msam2. The structural piece.
2. [In-process saga (with HTTP fallback)](#2-in-process-saga-with-http-fallback) — saga runs as an imported Python module in mimir's process by default. No subprocess, no port management, no health-check race. External saga via `SAGA_ENDPOINT` opt-in.
3. [Integration bench harness](#3-integration-bench--mimir-side-hooks-on-the-existing-longmemeval-harness) — end-to-end benches that exercise saga through mimir's actual call sites, complementing saga-only synthetic benches.
4. [muninnbot pin + transition note](#4-muninnbot-pin) — pin muninnbot's docker to a known msam2 source commit before archiving, so its build keeps working through the deprecation window.
5. [Doc cleanup](#5-doc-cleanup) — V0.4.md path refs, SPEC.md cross-refs, README, FUTURE_WORK §11 update.
6. [Rename: msam → saga](#6-rename-msam--saga) — the package is renamed as part of v0.5, in a focused follow-up commit on top of the merge. Touches the saga package's internal imports, mimir's `MsamClient` / config / env vars, and event type names.
7. [LLM unification — saga uses the `anthropic` SDK](#7-llm-unification--saga-uses-the-anthropic-sdk) — saga's 9 LLM call sites swap from raw `requests.post` to `anthropic.AsyncAnthropic.messages.create`. Mimir's plan-window utilization (v0.4 `rate_limits.py`) becomes the single source of truth for LLM consumption; saga's consolidation cron visible in the same window.

**Deferred to v0.6+:**
- Per-channel persona — design from v0.4's deferred list, ~50 LOC change in `load_core` to take channel_id. Independent of the merge work; pick up when feature priorities call for it.
- Temporal index for retrieval (FUTURE_WORK §3.1) — dominant probe-type miss in bench data; needs saga-side schema work that's much easier post-merge.
- Reply / repost graph (FUTURE_WORK §3.3) — weakest probe types; similar dependency pattern.
- Belief-revision / claims-audit reflection track (FUTURE_WORK §8.3 partial) — extends the v0.4 reflection skill once the operator and agent have settled patterns from a few weeks of running it.

## 2. In-process saga (with HTTP fallback)

**Goal:** new operator runs `mimir setup --home X && mimir run --home X` and gets a working agent + memory pair, no separate saga install ceremony, no subprocess to supervise. Saga runs in the same Python process as mimir by default. External saga deployments still work via `SAGA_ENDPOINT` override.

(All references in this section assume §6 has landed and the package is named `saga`. If reading post-merge but pre-rename, mentally substitute `msam` → `saga` and `MSAM_*` → `SAGA_*`.)

### Why in-process by default

Saga's actual deployment shape is 1:1 with mimir — one mimir, one saga, on the same host. The HTTP/JSON layer between them is overhead with no operational benefit:

- Every `MsamClient.query` today is encode→TCP→decode roundtrip. Pre_message_hook fires on every turn; this saves 5-50ms/call depending on payload.
- No port management, no health-check race, no supervisor restart-on-crash logic.
- Stack traces span both packages. `pdb` works across the boundary.
- `mimir run` is genuinely one process, one venv, one debugger.

The HTTP path stays as an explicit opt-out for operators who do want saga deployed separately (shared instance across multiple agents, scaling reasons, dev pointing at a staging saga).

### Adapter pattern

Mimir's existing `MsamClient` becomes a unified `SagaClient` interface with two implementations:

```python
# mimir/saga_client.py
class SagaClient(Protocol):
    async def query(self, query: str, **kw) -> dict: ...
    async def feedback(self, atom_ids: list[str], response_text: str, **kw) -> dict: ...
    async def end_session(self, session_id: str, summary: str, **kw) -> dict: ...
    async def store(self, content: str, **kw) -> dict: ...
    async def recent_session_boundaries(self, **kw) -> list[dict]: ...
    async def most_retrieved_atoms(self, **kw) -> list[dict]: ...

class _InProcessSaga(SagaClient):
    """Direct calls into saga.core. Saga's hot-path functions are sync
    (def, not async def), so we wrap each call in asyncio.to_thread to
    keep mimir's event loop responsive while saga does CPU-heavy work
    (embeddings, FTS5, vector ops)."""

    async def query(self, query: str, **kw) -> dict:
        from saga.core import hybrid_retrieve
        return await asyncio.to_thread(hybrid_retrieve, query, **kw)

    # ... other methods follow the same pattern.

class _HttpSaga(SagaClient):
    """Current implementation — HTTP via aiohttp. Used when
    SAGA_ENDPOINT points off-localhost (external saga deployment)."""

def make_saga_client(config: Config) -> SagaClient:
    if config.saga_endpoint and not _is_localhost(config.saga_endpoint):
        return _HttpSaga(config.saga_endpoint, config.saga_api_key)
    return _InProcessSaga()
```

Mimir's call sites (`agent.py`, `msamtools.py` → `sagatools.py` per §6) are unchanged — they all do `await self._saga.query(...)`. The implementation is a deployment choice.

### Why `to_thread` is the right wrapping

Saga's core functions are sync — `hybrid_retrieve`, `keyword_search`, `store_atom`, `mark_contributions`, etc. all `def`, not `async def`. The HTTP server today wraps them in FastAPI's async route handlers, which lets the sync work run on the event loop (and blocks it during heavy queries — they haven't hit that under load).

In-process integration uses `asyncio.to_thread` so saga's CPU-heavy work runs on a worker thread while mimir's event loop stays free. This is what saga's HTTP path effectively did via FastAPI's executor; making it explicit at the boundary is just better hygiene.

### Setup-time integration

`mimir setup` post-merge gains saga-aware behavior:

1. Detects whether `saga` is importable in the current Python env (always true in the workspace install).
2. Generates a `<home>/saga.toml` derived from `saga/saga.example.toml` with mimir-prod overrides:
   - `[storage].db_path = <home>/.mimir/saga.db` (sibling to `<home>/.mimir/index.db` — see "SQLite layout" below)
   - **`[retrieval].enable_contextual_rewrite = true`** — mimir already passes `context=` on every `/v1/query`; flipping the rewrite on means short referential queries ("yes, look for that") get resolved before retrieval. No-op when context is empty, so safe to leave on.
   - **`[triples].enable_extraction = true`** + **`[retrieval].enable_triple_augment_v2 = false`** — populate the triples table on consolidation so future query-intent-gated P41 work has data to read; don't enable P41 in retrieval today (per the post-fix bench: -3.7pp multi-session, -2.3pp temporal, both core mimir query types).
   - Otherwise inherits saga's canonical defaults (two_tier on, supersedes_demotion on, P12 query expansion on, confidence gating with `low` floor, P30 missing-ref pivot, spreading activation on). These are saga's `_DEFAULTS` per its `config.py`; the example.toml documents them.
3. Generates `SAGA_API_KEY=<random>` in mimir's `<home>/.env` and matching `[server].api_key` in the generated saga.toml. **The API key is unused in in-process mode** — it only matters if the operator later runs saga's HTTP server (e.g., for a separate dev tool, or if they switch to external-saga mode). Generating it up-front means flipping later doesn't require a setup re-run.
4. Does NOT write `SAGA_ENDPOINT` by default — its absence is what signals in-process mode. If the operator wants to point at an external saga, they set `SAGA_ENDPOINT=https://my-saga.internal/` in `.env` themselves.

Existing keys / configs preserved on re-run, matching the existing MIMIR_API_KEY behavior. Existing `<home>/saga.toml` files preserved — operator can edit freely without setup clobbering.

The contextual_rewrite LLM call uses saga's `[retrieval_v2]` LLM config (falls back to `[llm]`). Mimir's setup should make sure at least one LLM is configured for saga if rewrite is on; otherwise the rewrite logs a warning and falls through to the original query (still functional). After §7 lands, this LLM is the same Anthropic key mimir uses — single config point.

### Health check + visibility

- Startup: `_InProcessSaga.__init__` does a no-op call (`saga.get_stats()`) and logs the result. Failure (e.g., DB unreachable, embedding model fails to load) surfaces immediately rather than at first query.
- New event types: `saga_health_ok`, `saga_health_failed` — a single periodic health check per heartbeat (60s) with the saga DB, surfaced in algedonic block via `feedback.py` rules so the agent sees if saga's DB has gone bad.
- `mimir stats` gains a saga line: in-process vs HTTP mode, last health check, atom count, DB size on disk.

### SQLite layout

Mimir's existing `<home>/.mimir/index.db` (file_search chunks + embeddings) and saga's `<home>/.mimir/saga.db` (atoms, access_log, triples, etc.) live side-by-side in the same directory. **Separate files, not a shared database.** Reasons:

- **Lock contention.** SQLite is single-writer per file. Saga's consolidation pass writes for several minutes; sharing a file would block mimir's per-turn reindexes.
- **Different hot paths.** Mimir's file_search is per-turn, latency-sensitive. Saga's atom writes are sporadic but heavy. Independent lock granularity matters.
- **Different lifecycles.** `mimir setup` can blow away `index.db` and rebuild from `memory/`/`state/` files. Saga's atoms can't be regenerated — they're the canonical store.
- **Schema migration blast radius.** Saga has its own schema migrations; mimir's search.py has its own. Separating files means a saga schema bug can't corrupt mimir's index.

Same directory gets the operational wins (one backup target, one mount point) without the entanglement.

(Future-optional: shared embedding cache as a third file, e.g. `embed_cache.db` keyed by `(provider, model, content_hash) → blob`. Useful if mimir + saga ever standardize on one provider. Not blocking; not in v0.5.)

### Opt-out paths

- `SAGA_ENDPOINT=https://...` (non-localhost) → mimir uses `_HttpSaga` against the external endpoint. In-process saga isn't loaded.
- `mimir setup --no-saga` → setup skips saga.toml generation. Operator must point `SAGA_ENDPOINT` at something external before running.

## 3. Integration bench — mimir-side hooks on the existing LongMemEval harness

**Goal:** make saga's existing LongMemEval bench measure retrieval quality *through* mimir's actual call sites, not just saga-in-isolation. So when v0.6+ retrieval changes ship, we know whether the win is still a win after mimir's caching, contextual-rewrite, credit-pass, and session-boundary lifecycle interact with it.

### What's already there (post-merge inheritance)

saga's bench at `saga/saga/benchmarks/longmemeval/` (inside the Python package, runnable standalone) is mature:

- LongMemEval `S` (500 questions). Reader: MiniMax-M2.7. Judge: gpt-4o. Consolidation: gpt-5.4-nano.
- `run_eval.py` orchestrates per-question sqlite DBs (created in a work dir, destroyed after each question unless `--keep-dbs`).
- Variance characterized: ±1.3pp overall (N=500), per-subtype noise floors documented in `BENCHMARK-RESULTS.md`. Don't ship on sub-noise-floor wins.
- Canonical baseline: `msam_p30_minimax_v3` at 0.784. Best to date: `msam_p9_minimax_v2` at 0.796 (older codebase) and `msam_p12_canon_v1` at 0.792 (P12 query expansion, recently shipped).

This harness is the source of truth for retrieval quality. Don't replace it; extend it.

### What's invisible to saga-in-isolation benches

LongMemEval today drives saga directly via the Python API or `/v1/query`. It doesn't exercise:

- **Cache interaction.** Mimir's prompt cache changes which atoms get re-fetched turn-to-turn; whether a retrieval improvement compounds with cache hits or fights them is invisible at the saga layer.
- **Contextual rewrite firing rate.** With `enable_contextual_rewrite = true` (now the v0.5 default), short referential queries get rewritten via LLM. Bench shows the rewrite firing on the appropriate fraction of LongMemEval probes, the rewrites' quality, and the resulting end-to-end delta.
- **Mark-contributions credit pass.** Mimir's `mark_contributions` after each agent reply tags atoms that contributed; that signal feeds future retrieval ranking. End-to-end bench shows whether the credit signal is actually helping over multi-question runs.
- **Session boundaries.** Which `session_id` is in flight (per the `e88d458` schema migration) determines the credit-pass scope. End-to-end bench measures within-session vs. cross-session recall.
- **`recent_session_boundaries` / `most_retrieved_atoms` (P45) usage.** These are mimir-side prompt sections. Whether populating them improves end-to-end task accuracy is a question the saga-only bench can't answer.

### Harness shape

The integration bench lives at the **workspace root**, not inside `saga/`, so the saga package stays mimir-independent:

```
mimir/                                  (workspace root)
├── saga/saga/benchmarks/longmemeval/   # existing — saga-direct, unchanged.
│   ├── run_eval.py                     #   Imports saga only.
│   └── ...                             #   Runnable standalone:
│                                       #     cd saga && uv run python -m \
│                                       #       saga.benchmarks.longmemeval.run_eval
│
└── benchmarks/longmemeval_via_mimir/   # NEW — cross-package integration.
    ├── runner.py                       #   Imports both mimir and saga.
    ├── route.py                        #   Boots mimir's BenchBridge agent
    ├── score.py                        #   in-process; saga runs in-process via
    │                                   #   _InProcessSaga (no subprocess); adapts
    │                                   #   LongMemEval probes → channel messages.
    │                                   #   Reuses saga's scoring via score.py.
    └── README.md
```

Why outside `saga/`: if `longmemeval_via_mimir/runner.py` lived under the saga package and imported mimir, then importing the saga package transitively would fail in any venv where mimir isn't installed. Standalone saga ops would silently break. Living at workspace root makes the dependency direction obvious: integration benches use both packages; neither package depends on the integration tree.

The runner uses mimir's existing `BenchBridge` (designed for this) so bench probes hit the same dispatch path as a live Slack/Discord message — pre_message_hook fires (queries saga), the agent reads the assembled prompt (with context, plan windows, etc.), assembles its response, post_message_hook fires (mark_contributions). All visible.

### Path-frequency telemetry

The handoff doc points at saga's `saga.retrieval` Python logger (renamed from `msam.retrieval` in §6) which emits one tagged line per LLM-path decision (`path=combined`, `path=rewrite_only`, `path=hyde_gated fired=...`, `path=subatom_beam atoms=N`, `path=triple_augment_v2 atoms=N`). v0.5 wires this into mimir's logging setup so:

- The bench's stderr capture shows path frequencies per LongMemEval run.
- `mimir stats` gains a path-frequency snapshot for production use.
- The algedonic block surfaces a warning if (e.g.) HyDE keeps firing in a config that didn't enable it — a misconfiguration smoke alarm.

**Goal:** the package becomes `saga` end-to-end. Decided after a session of brainstorming; rationale in V0.5.md commit history. ### Commit A: rename inside the saga package

## 7. LLM unification — saga uses the `anthropic` SDK

**Goal:** saga makes its LLM calls (consolidation, contextual rewrite, triples extraction, HyDE if flagged) via the `anthropic` Python SDK rather than raw `requests.post` to OpenAI-compatible chat-completions endpoints. Mimir's plan-window utilization (v0.4 `rate_limits.py`) becomes the single source of truth for LLM consumption — saga's consolidation cron suddenly visible in the same window the agent reports against.

### Why anthropic SDK, not claude-agent-sdk

| | `claude-agent-sdk` | `anthropic` SDK |
|---|---|---|
| What it is | Wraps the Claude Code CLI subprocess; built for multi-turn agent loops with tool use, hooks, MCP servers | Direct Messages API client; single-call request/response |
| Saga's LLM calls | Single-turn extractions and rewrites (no tool use, no multi-turn) | **Right shape.** |
| Decision | No — over-engineered for saga's use | Yes — use this one |

mimir uses claude-agent-sdk because mimir IS the agent loop. Saga doesn't have an agent loop to wrap — it has discrete LLM-utility calls. Different tool for a different job.

### What saga does today

Per the live `~/projects/odin/msam-hindsight-ideas/` worktree: 9 LLM call sites across 6 files. All use raw `requests.post` to OpenAI-compat chat-completions endpoints. Per-subsystem config (`[consolidation]`, `[triples]`, `[retrieval_v2]`) overrides `llm_url`, `llm_model`, `api_key_env`; falls back to top-level `[llm]`. Standard pattern:

```python
import requests
r = requests.post(
    llm['url'],
    headers={"Authorization": f"Bearer {llm['api_key']}"},
    json={"model": llm['model'],
          "messages": [{"role": "user", "content": prompt}],
          "max_completion_tokens": 1500},
    timeout=llm['timeout'],
)
msg = r.json()["choices"][0]["message"]
text = msg.get("content") or msg.get("reasoning") or ""
```

Call sites:

- `core.py:1565` — contextual rewrite
- `core.py:1672` — HyDE (off by default per bench)
- `core.py:1772` — `_resolve_query_and_hypothetical` (combined rewrite+HyDE LLM call when both flags fire)
- `consolidation.py:693` — cluster summary + triple extraction (the main consolidation LLM)
- `triples.py:268` — store-time triple extraction (deprecated; consolidation-only is the active path per recent cleanup)
- `retrieval_v2.py:313` — single-tier path; bypassed when `two_tier=true` (canonical default)
- `subatom.py:578` — sentence-level extraction for subatom beam (off by default per bench)
- `annotate.py:201` — `llm_annotate` for atom storage

### Migration shape

Replace the raw HTTP wire-format with `anthropic.AsyncAnthropic`:

```python
from anthropic import AsyncAnthropic

# saga/_llm.py — new module, single LLM client wrapper
_client: AsyncAnthropic | None = None

def _get_client() -> AsyncAnthropic:
    global _client
    if _client is None:
        _client = AsyncAnthropic(api_key=os.environ["ANTHROPIC_API_KEY"])
    return _client

async def call_llm(
    *, model: str, prompt: str, max_tokens: int = 1500,
    temperature: float = 0.3, system: str | None = None,
) -> str:
    client = _get_client()
    msg = await client.messages.create(
        model=model,
        max_tokens=max_tokens,
        temperature=temperature,
        system=system or "",
        messages=[{"role": "user", "content": prompt}],
    )
    return msg.content[0].text if msg.content else ""
```

Each call site becomes:

```python
# Before:
import requests
r = requests.post(llm['url'], json={...}, ...)
text = r.json()["choices"][0]["message"]["content"]

# After:
from saga._llm import call_llm
text = await call_llm(
    model=llm['model'], prompt=prompt,
    max_tokens=1500, temperature=0.3,
)
```

Some call sites are inside sync functions (saga's whole core is sync — see §2). They use `asyncio.run()` or get refactored to be async-callable. Pragmatic answer: keep the sync wrapper too:

```python
def call_llm_sync(**kwargs) -> str:
    return asyncio.run(call_llm(**kwargs))
```

…and use the sync version everywhere saga's existing sync code path lives. The `to_thread` wrapping at mimir's boundary (§2) means mimir's event loop isn't blocked even if saga's LLM call is sync internally.

### Provider knob (back-compat for bench parity)

Bench results today are characterized against `gpt-5.4-nano` on consolidation (per the post-fix `msam_p30_canon_v4` baseline at 0.774). Switching mid-bench-cycle invalidates the comparison. Keep the OpenAI-compat path as a fallback gated by config:

```toml
[llm]
provider = "anthropic"  # or "openai_compat"
```

When `provider = "anthropic"` (the new default for production deployments), the new `_llm.py` path runs. When `provider = "openai_compat"`, saga keeps its existing `requests.post` flow — unchanged. Bench harness ships with `openai_compat` so historical comparability is preserved until the team runs an Anthropic-baseline bench and decides to flip the default.

### Cost / latency / quota implications

- **Cost.** gpt-5.4-nano is cheap (~$0.0001/1K input tokens). Claude Haiku 4.5 is competitive (~$0.001/1K input, $0.005/1K output) but ~10× the input rate. For consolidation specifically (~50 clusters × ~1500 prompt tokens × ~500 output = ~100K total tokens per run), that's $0.10/run on Claude vs $0.01/run on gpt-5.4-nano. Multiplied by daily/weekly cron, real but bounded.
- **Latency.** Comparable. Anthropic API and OpenAI-compatible APIs are both sub-second for short prompts.
- **Plan-window quota unification.** This is the substantive operational win. Today saga's LLM calls are invisible to mimir's cost-tracking — they hit a separate (OpenAI / Nvidia NIM / etc.) account. After §7, saga's calls hit the same Anthropic plan window mimir's main agent reports against. Mimir's "Resource usage" prompt block sees the full picture: "this turn cost $X, but a consolidation pass also fired and ate Y% of the 5h window."
