# Mimir v0.5 — workspace monorepo + bundled MSAM

**Goal:** end the cross-repo coordination tax between mimir and msam2 by merging the MSAM source tree into mimir as a workspace package, then close the operator-UX gap so `mimir setup && mimir run` brings up a working agent + memory pair without separate MSAM install ceremony.

**Frame:** v0.4 made mimir self-aware about resources (cost, plan windows, subagent spend). The next blocker on velocity is structural — every PR that touches the mimir/MSAM boundary today is a two-repo coordination dance, and every new operator who installs mimir hits a wall when MSAM isn't running. v0.5 makes both go away.

## Pieces (high-leverage to small)

1. [Workspace monorepo merge](#1-workspace-monorepo-merge) — filter-repo msam2 into mimir at `/msam/`, set up uv workspace, archive msam2. The structural piece.
2. [Bundled MSAM lifecycle](#2-bundled-msam-lifecycle) — `mimir setup` generates a shared API key into both configs, `mimir run` supervises MSAM as a child process, opt-out via `MIMIR_SPAWN_MSAM=false` for external-MSAM deployments.
3. [Integration bench harness](#3-integration-bench-harness) — end-to-end benches that exercise MSAM through mimir's actual call sites, complementing MSAM-only synthetic benches.
4. [muninnbot pin + transition note](#4-muninnbot-pin) — pin muninnbot's docker to a known msam2 source commit before archiving, so its build keeps working through the deprecation window.
5. [Doc cleanup](#5-doc-cleanup) — V0.4.md path refs, SPEC.md cross-refs, README, FUTURE_WORK §11 update.

**Deferred to v0.6+:**
- Per-channel persona — design from v0.4's deferred list, ~50 LOC change in `load_core` to take channel_id. Independent of the merge work; pick up when feature priorities call for it.
- Temporal index for retrieval (FUTURE_WORK §3.1) — dominant probe-type miss in bench data; needs MSAM-side schema work that's much easier post-merge.
- Reply / repost graph (FUTURE_WORK §3.3) — weakest probe types; similar dependency pattern.
- Belief-revision / claims-audit reflection track (FUTURE_WORK §8.3 partial) — extends the v0.4 reflection skill once the operator and agent have settled patterns from a few weeks of running it.

### History preservation

`git filter-repo` extracts msam2's tracked source + canonical docs into a temporary repo (everything renamespaced under `msam/`), then merge as a new tree.

Verified inputs from the live `~/projects/odin/msam-hindsight-ideas/` worktree:

- 4.5M total tracked content. The `data/`, `external/`, `results/` directories are filesystem symlinks to the sibling msam/ checkout; their bytes aren't in git, so no bench-data filtering is needed.
- Bench harness lives at `msam/benchmarks/` (inside the Python package); `--path msam/` covers it.
- Canonical operational docs at root: `BENCHMARK-RESULTS.md`, `BENCHMARKS.md`, `CONTROL-FLOW.md`, `HINDSIGHT-IDEAS.md`, `NEXT-EXPERIMENTS.md`, `SPEC.md`, `README.md`. All come along.
- `MIMIR-HANDOFF.md` is a transition snapshot — explicitly excluded; once the merge lands, the live tree is the source of truth.
- 125 commits ahead of msam2 main on the hindsight-ideas branch. All come along.

```bash

#    --depth=1; full history is the point).

#    aren't tracked, so no extra filtering needed for them.

    --path msam/ \
    --path BENCHMARK-RESULTS.md \
    --path BENCHMARKS.md \

    --path HINDSIGHT-IDEAS.md \

    --path SPEC.md \

git remote add msam-import /tmp/msam-export
git fetch msam-import
git merge --allow-unrelated-histories msam-import/hindsight-ideas \
    -m "Import MSAM source under msam/ via filter-repo

branch (125 commits beyond main). Symlinked data/external/results
were never in git history and stay external.

# 4. Validate: msam/ has full file history, no surprises imported.

du -sh msam/

# 5. Bring main over too if you want it preserved as a tag/branch.
#    Optional — hindsight-ideas already contains main's history.
git fetch msam-import refs/heads/main:refs/tags/msam-main-pre-merge
```

### Workspace config

Root `pyproject.toml`:

```toml
[tool.uv.workspace]
members = ["mimir", "msam"]

[tool.uv.sources]
mimir = { workspace = true }
msam = { workspace = true }
```

Each package keeps its own `pyproject.toml` with its own deps, version, etc. `uv sync` at root installs both editable; either can still publish to PyPI independently.

**Versioning:** single version line — mimir's git tag IS msam's version. Both packages' `pyproject.toml` versions track the tag. Trade-off: msam-only changes don't get distinct version numbers, but msam2 had no real PyPI consumers, so this is theoretical cost only.

## 2. Bundled MSAM lifecycle

**Goal:** new operator runs `mimir setup --home X && mimir run --home X` and gets a working agent + memory pair, no separate MSAM install ceremony. External MSAM deployments still work via `MSAM_ENDPOINT` override.

### Setup-time integration

`mimir setup` post-merge gains MSAM-aware behavior:

1. Detects whether `msam` is importable in the current Python env (it always will be in the workspace install).
2. Generates a single API key (`secrets.token_urlsafe(32)`).
3. Writes the key to mimir's `<home>/.env` as `MSAM_API_KEY=...`.
4. Writes a generated `<home>/msam.toml` derived from `msam/msam.example.toml` with mimir-prod overrides:
   - `[server].api_key = <generated>`
   - `[server].port = <auto-pick free port>` (default 3002, fall back to next free)
   - `[storage].db_path = <home>/.mimir/msam.db`
   - **`[retrieval].enable_contextual_rewrite = true`** — mimir already passes `context=` on every `/v1/query`; flipping the rewrite on means short referential queries ("yes, look for that") get resolved before retrieval. No-op when context is empty, so safe to leave on.
   - Otherwise inherits MSAM's canonical defaults (two_tier on, supersedes_demotion on, P12 query expansion on, confidence gating with `low` floor, P30 missing-ref pivot, spreading activation on). These are MSAM's `_DEFAULTS` per its `config.py`; the example.toml documents them.
5. Writes mimir's `MSAM_ENDPOINT=http://localhost:<chosen port>` matching.

Existing keys preserved on re-run, like the existing MIMIR_API_KEY behavior. Existing `<home>/msam.toml` files preserved too — operator can edit freely without setup clobbering.

The contextual_rewrite LLM call uses MSAM's `[retrieval_v2]` LLM config (falls back to `[llm]`). Mimir's setup should make sure at least one LLM is configured for MSAM if rewrite is on; otherwise the rewrite logs a warning and falls through to the original query (still functional).

### Run-time supervisor

`mimir run` post-merge gains an MSAM child-process supervisor when `MIMIR_SPAWN_MSAM=true` (default `true` if `msam` is importable; default `false` if not).

```python
# mimir/msam_supervisor.py — new module
class MsamSupervisor:
    """Forks msam.server, monitors, restarts on crash with backoff,
    pipes stderr into mimir's logger, graceful-shuts on parent signal."""

    async def start(self) -> None:
        # spawn, wait for /v1/health to return 200
        ...

    async def _monitor(self) -> None:
        # poll process; on exit, restart with exponential backoff
        # cap at N restarts in M minutes; past that, log critically
        # and stop trying (operator's problem)
        ...

    async def stop(self) -> None:
        # SIGTERM, wait for graceful shutdown, SIGKILL if needed
        ...
```

Supervisor lifecycle hooks into mimir's existing aiohttp app startup/shutdown handlers in `server.py`.

### Health check + visibility

- Startup log: `"MSAM supervisor: spawned PID 12345 on port 3002, /v1/health OK"` or `"MSAM_ENDPOINT=external; not spawning"`.
- New event types: `msam_supervisor_spawned`, `msam_supervisor_restarted`, `msam_supervisor_exited`. Surfaced in algedonic block via `feedback.py` rules so the agent sees if MSAM is unhealthy.
- `mimir stats` gains an MSAM line: process status, uptime, restart count.

### Opt-out paths

- `MIMIR_SPAWN_MSAM=false` — disable spawn entirely.
- `MSAM_ENDPOINT` set to non-localhost → automatically disable spawn (operator pointing at external MSAM).
- `mimir setup --no-msam` flag → setup skips MSAM config generation.

### Tests

- `tests/test_msam_supervisor.py`:
  - Spawn happy path (mock subprocess)
  - Health-check timeout → supervisor surfaces error
  - Crash + restart with backoff
  - Graceful shutdown on parent stop
  - External MSAM endpoint → supervisor doesn't spawn

### Effort

- `MsamSupervisor` module: ~150 LOC
- Setup integration: ~80 LOC (key generation, msam.toml writer)
- Tests: ~150 LOC (subprocess mocking is the bulk)
- `feedback.py` rule additions: ~10 LOC

**Total: ~390 LOC. ~2 focused days.**

## 3. Integration bench — mimir-side hooks on the existing LongMemEval harness

**Goal:** make MSAM's existing LongMemEval bench measure retrieval quality *through* mimir's actual call sites, not just MSAM-in-isolation. So when v0.6+ retrieval changes ship, we know whether the win is still a win after mimir's caching, contextual-rewrite, credit-pass, and session-boundary lifecycle interact with it.

### What's already there (post-merge inheritance)

MSAM's bench at `msam/benchmarks/longmemeval/` is mature:

- LongMemEval `S` (500 questions). Reader: MiniMax-M2.7. Judge: gpt-4o. Consolidation: gpt-5.4-nano.
- `run_eval.py` orchestrates per-question sqlite DBs (created in a work dir, destroyed after each question unless `--keep-dbs`).
- Variance characterized: ±1.3pp overall (N=500), per-subtype noise floors documented in `BENCHMARK-RESULTS.md`. Don't ship on sub-noise-floor wins.
- Canonical baseline: `msam_p30_minimax_v3` at 0.784. Best to date: `msam_p9_minimax_v2` at 0.796 (older codebase) and `msam_p12_canon_v1` at 0.792 (P12 query expansion, recently shipped).

This harness is the source of truth for retrieval quality. Don't replace it; extend it.

### What's invisible to MSAM-in-isolation benches

LongMemEval today drives MSAM directly via the Python API or `/v1/query`. It doesn't exercise:

- **Cache interaction.** Mimir's prompt cache changes which atoms get re-fetched turn-to-turn; whether a retrieval improvement compounds with cache hits or fights them is invisible at the MSAM layer.
- **Contextual rewrite firing rate.** With `enable_contextual_rewrite = true` (now the v0.5 default), short referential queries get rewritten via LLM. Bench shows the rewrite firing on the appropriate fraction of LongMemEval probes, the rewrites' quality, and the resulting end-to-end delta.
- **Mark-contributions credit pass.** Mimir's `mark_contributions` after each agent reply tags atoms that contributed; that signal feeds future retrieval ranking. End-to-end bench shows whether the credit signal is actually helping over multi-question runs.
- **Session boundaries.** Which `session_id` is in flight (per the `e88d458` schema migration) determines the credit-pass scope. End-to-end bench measures within-session vs. cross-session recall.
- **`recent_session_boundaries` / `most_retrieved_atoms` (P45) usage.** These are mimir-side prompt sections. Whether populating them improves end-to-end task accuracy is a question the MSAM-only bench can't answer.

### Harness shape

Two integration layers, both extending what's already at `msam/benchmarks/`:

```
msam/benchmarks/
├── longmemeval/                # existing — MSAM-direct
│   ├── run_eval.py             # canonical orchestrator
│   ├── ...
└── longmemeval_via_mimir/      # NEW — same probes, routed through mimir's
    │                           # BenchBridge so retrieval goes through the
    │                           # whole pre/post-message hook stack
    ├── runner.py               # spawns supervised MSAM + boots mimir's
    │                           # BenchBridge agent in-process
    ├── route.py                # adapts LongMemEval probes → bench channel
    │                           # messages → captures the agent's response
    ├── score.py                # adapts existing LongMemEval scoring
    └── README.md               # how to run, what it measures
```

The runner uses mimir's existing `BenchBridge` (designed for this) so bench probes hit the same dispatch path as a live Slack/Discord message — pre_message_hook fires (queries MSAM), the agent reads the assembled prompt (with context, plan windows, etc.), assembles its response, post_message_hook fires (mark_contributions). All visible.

### Path-frequency telemetry

The handoff doc points at MSAM's `msam.retrieval` Python logger which emits one tagged line per LLM-path decision (`path=combined`, `path=rewrite_only`, `path=hyde_gated fired=...`, `path=subatom_beam atoms=N`, `path=triple_augment_v2 atoms=N`). v0.5 wires this into mimir's logging setup so:

- The bench's stderr capture shows path frequencies per LongMemEval run.
- `mimir stats` gains a path-frequency snapshot for production use.
- The algedonic block surfaces a warning if (e.g.) HyDE keeps firing in a config that didn't enable it — a misconfiguration smoke alarm.

### Reading list pointer

Per MSAM-Claude's first-30-minutes guide (preserved in HINDSIGHT-IDEAS.md / NEXT-EXPERIMENTS.md post-merge):

1. `msam/msam/config.py:_DEFAULTS` (~lines 63-200) — canonical config keys.
2. `msam/msam/server.py` lines 1-150 — request/response contract.
3. `msam/BENCHMARK-RESULTS.md` summary table — what's been tried.
4. `msam/msam/core.py:hybrid_retrieve` (~line 1830) — the canonical retrieval entry point that composes pre-retrieval LLM dispatch, the two-pathway retrieve+keyword passes, optional graph/HyDE/subatom/triple_augment_v2, RRF fusion, two-tier split with P30 missing-atom pull-in, P9 evidence boost.

### What this unlocks for v0.6+

- **Temporal-index work** (FUTURE_WORK §3.1) becomes end-to-end measurable.
- **Reranker work** (FUTURE_WORK §7.5) gets the same treatment.
- **Open MSAM experiments P40, P42, P44** (per the handoff doc) can be benched through mimir to see whether MSAM-side wins survive the integration.

## Bench predictions

v0.5 is structural — no direct bench-quality impact expected. The win shows up indirectly:

- §3 integration bench harness exists and is exercised. Future v0.6+ retrieval changes get measured end-to-end.
- §2 bundled MSAM removes a class of "MSAM is down" failures that today degrade silently into empty `recent_session_boundaries` / `most_retrieved_atoms` returns.
- Operationally: time-to-first-working-agent for a new operator drops from "install MSAM, configure msam.toml, start MSAM, sync API key, edit mimir's .env, start mimir" to "install mimir, run setup, run run."
